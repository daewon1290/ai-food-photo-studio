# 개발 로그

프로젝트의 주요 결정 사항과 방향 전환을 시간순으로 정리한다.

---

## Phase 1 — 초기 MVP (commit: 41a1e15)

**구현 내용**
- Next.js App Router 기반 프로젝트 구조
- OpenAI `gpt-image-1` `images.edit` API 연동
- 사진 업로드 → 프롬프트 생성 → 이미지 반환 기본 흐름

**초기 템플릿 구조**
- 8개 템플릿, 감성 설명 문장 기반 프롬프트
- 문제: 템플릿이 달라도 비슷한 이미지로 수렴

---

## Phase 2 — 템플릿 구조화

**문제 인식**
- 감성 설명 프롬프트는 AI가 무시하거나 비슷하게 해석함
- 템플릿 간 차이가 결과물에서 보이지 않음

**해결 방향**
- 템플릿 = 촬영 규칙 구조체로 재정의
- `foodRatio`, `cameraAngle`, `tableSurface`, `sceneProps` 등 명시적 필드로 분리
- `buildPrompt.ts`가 규칙 → 프롬프트로 조립하는 구조 도입

**결과**
- 템플릿 간 결과물 차이 명확해짐
- 새 템플릿 추가 시 규칙만 채우면 됨

---

## Phase 3 — A/B 시안 구조 도입

**배경**
- 사용자는 보통 여러 옵션이 있어야 고를 수 있음
- "음식 중심"과 "소품 풍성" 두 니즈가 명확히 다름

**설계 결정**
- 시안 A: `foodRatioA` + 소품 0~1개 → 깔끔한 메뉴사진
- 시안 B: `foodRatioB` + sceneProps 3~5개 → 광고 스타일
- A/B는 독립 API 호출 — "A와 달리" 같은 상호 참조 지시 불가

**구현**
- `buildStudioPrompt` / `buildPremiumPrompt` 분리
- A/B 2회 병렬 호출 (`Promise.all`)
- `PhotoComparison` 컴포넌트로 나란히 비교 UI

## Phase 4 — Flux 모델 추가

**배경**
- GPT는 어두운 배경 연출에서 음식이 배경에 묻히는 문제
- 다크 프리미엄, 바 분위기 등은 Flux가 더 나은 결과

**문제 발견**
- Flux Kontext Pro는 긴 프롬프트 후반부를 무시
- GPT용 구조화 프롬프트를 그대로 쓰면 템플릿이 달라도 동일한 이미지 수렴

**해결**
- Flux 전용 압축 프롬프트 빌더 (`buildFluxStudioPrompt`, `buildFluxPremiumPrompt`) 추가
- 차별화 핵심(스타일·소품)을 앞에 배치, 1,200자 이내 목표
- `template.preferredModel`로 라우팅 — 각 템플릿이 자신에게 맞는 모델 지정

## Phase 5 — 음식 보존 레이어 정교화

**문제**
- AI가 원본 음식을 재창조하거나 변형하는 사고 발생
  - 먹태(건조 명태 채) → 감자튀김으로 변환
  - 음료 → 빙수·아이스크림으로 변환
  - 그릇 교체 (원본 냄비 → 다크 냄비)
  - 단품 사진에 반찬 자동 추가

**해결**
- `FOOD_IDENTITY_GUARDS` 도입 — 먹태, 음료 등 특정 변형 패턴 명시 금지
- 카테고리별 질감 보존 프롬프트 (`getMenuTextureNote`) — 튀김, 떡볶이, 음료
- 경고 UI (`noticeKo`) — 변형 위험이 높은 템플릿에 사용자 안내

## Phase 6 — 템플릿 확장 (8개 → 16개)

**추가된 템플릿**
- `hot-soup` — 뜨거운 국물·스팀형 (스팀이 핵심 비주얼)
- `cafe-drink` — 카페 음료형 (투명 글라스 레이어 보존)
- `fine-dining` — 파인다이닝 단품형 (넓은 여백)
- `cocktail-bar` — 칵테일·하이볼 바형
- `white-dessert` — 화이트 미니멀 디저트형
- `bbq-grill` — 바비큐·그릴형 (실험 템플릿)

- `cutout-solid` — 누끼·단색 배경형

- `bbq-grill`처럼 변형 위험이 높은 템플릿은 출시 기본 목록에서 제외 가능

## Phase 7 — 레퍼런스 이미지 지원

**배경**
- 특정 템플릿은 텍스트 프롬프트만으로 스타일 전달에 한계
- 실제 스타일 사진을 레퍼런스로 주면 더 정확한 결과

**구현**
- `template.referenceImagePath`로 레퍼런스 이미지 지정
- B 시안 요청 시에만 레퍼런스 첨부 (A는 흰 배경 공통 스펙)
- `images.edit`에 `[음식사진, 레퍼런스]` 배열 전달

## 실험 중인 접근법 (결론 미정)

### 마스크 기반 편집 (`/api/generate-preserve`)
- 음식 영역을 마스크로 보호 후 주변만 편집
- 음식 보존은 좋지만 배경 자연스러운 블렌딩이 어려움

### 원본 픽셀 합성 (`/api/generate-composite`)
- AI가 배경만 생성 후 원본 음식 픽셀을 합성
- Photoroom 배경 제거 API 의존성 필요

### Photoroom + AI 배경 합성 (`/api/ai-background-compose`)
- Photoroom으로 배경 제거 → GPT로 빈 배경 생성 → sharp 합성
- 소요 시간 20~40초로 현재 기본 방식 대비 느림

## 알려진 한계 및 수용 결정

| 현상 | 원인 | 대응 |
|---|---|---|
| gpt-image-1이 구도를 크게 변경하지 못함 | 모델 구조적 한계 | 목표를 "소품·조명 강화"로 재정의, 구도 대변환은 포기 |
| Flux 긴 프롬프트 무시 | 토큰 제한 | 압축 프롬프트 별도 제작 |
| A/B 차이가 약한 경우 | 모델이 소품 지시를 무시 | 소품 수를 줄이고 가장 임팩트 있는 것만 남김 |
| 손동작 생성 왜곡 | AI 손 생성 일반적 한계 | 실험적 기능으로 분류, 경고 UI 추가 |
