# 비용 관리 원칙

## API 비용 구조

### OpenAI gpt-image-1

`images.edit` API, `quality: 'high'`, `size: '1024x1024'` 기준.

- 버튼 1회 클릭 = API 호출 **2회** (A 시안 + B 시안 병렬)
- 고품질 이미지는 단가가 높다. 테스트 전 반드시 예상 호출 횟수 계산.

### fal.ai Flux Kontext Pro

`fal-ai/flux-pro/kontext` 모델.

- 버튼 1회 클릭 = API 호출 **2회** (A 시안 + B 시안 병렬)
- GPT 대비 단가는 낮지만, 템플릿 테스트 시 여전히 누적 비용에 주의.

## 테스트 원칙

**한 번에 다량 생성 금지.** 이미지 생성은 텍스트 API와 달리 호출당 비용이 크다.

테스트 전 체크리스트:
- [ ] 이번 테스트에서 API가 몇 회 호출되는지 확인
- [ ] 스크립트 루프가 있으면 반복 횟수 확인
- [ ] 기본은 1~3장 소량 테스트 후 품질 확인
- [ ] 전체 16개 템플릿 순회 테스트는 32회 호출 — 사전 승인 필요

## 스크립트 실행 전 주의사항

`scripts/` 디렉토리의 테스트 스크립트는 각각 여러 장을 생성할 수 있다.

| 스크립트 | 예상 호출 수 |
|---|---|
| `test-ab-split.ts` | 템플릿 수 × 2 |
| `test-premium-5templates.ts` | 5 × 2 = 10회 |
| `generate-template-covers.ts` | 템플릿당 1회 — 16회 이상 가능 |
| `test-routing-v1.ts` | 라우팅 테스트 대상 수 × 2 |

스크립트 실행 전 코드를 열어서 루프 범위를 먼저 확인한다.

## 개발 환경 절약 팁

1. **카테고리 없이 테스트** — 카테고리는 선택사항이고 API 호출 횟수에 영향 없음
2. **단일 템플릿 먼저** — 새 프롬프트 변경 사항은 가장 대표적인 1개 템플릿으로 검증
3. **Flux 우선 테스트** — GPT보다 단가 낮음. 품질 방향성 먼저 확인 후 GPT 전환
4. **에러 재현 시 단일 시안** — `promptB` 없이 A만 보내거나 `promptA`만 수정해서 테스트

## 프로덕션 고려사항

- 사용자당 일일 생성 횟수 제한 고려 (현재 미구현)
- `429` 에러 핸들링은 `generate-photo/route.ts`에 구현됨
- OpenAI 결제 한도 도달 시 `400` billing 에러로 구분하여 사용자에게 안내
